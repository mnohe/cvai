= ADR-003: Frontend interactivity

Status: Accepted +
Date: 2026-05-19

== Context

ADR-002 established Python + FastAPI + Jinja2 as the web application stack with server-side rendering.
The application returns complete HTML from the server; the browser renders it.

One interactive requirement exists beyond basic navigation and form submission: whilst a background operation runs (LLM ingestion, role re-assessment), the operation log must update in the browser without a full page reload.
The user needs to see progress and know when the operation completes or fails.

All other interactions (recording a status update, downloading an artefact, reviewing a task) are single request/response cycles that do not require in-page updates.

A secondary constraint: forms must remain functional without JavaScript as a baseline.
Progressive enhancement is preferable to a hard JavaScript dependency for actions that do not require live feedback.

== Alternatives considered

=== Full SPA (React, Vue, Svelte)

A single-page application framework would solve the polling requirement but imposes a build pipeline, a separate API layer, and client-side state management for a problem that requires one live-updating section on one page.
The overhead is disproportionate to the requirement.

=== Alpine.js

Alpine.js adds declarative reactivity without a build step and is well-suited to forms with client-side state.
It does not simplify the server-polling pattern, which still requires a fetch loop written in JavaScript.

=== Vanilla JavaScript

A small interval-based fetch loop is sufficient for the polling requirement and adds no library dependencies.
It replicates what a library would provide, adds maintenance burden without a framework contract, and does not generalise to future interactive requirements without growing in complexity.

=== Server-Sent Events

SSE is an appropriate long-term mechanism for streaming operation log lines from the server to the browser: the server pushes events; the client receives them on an open connection without polling.
An asyncio-native server supports SSE natively once the threading/asyncio migration is complete.
SSE reduces per-line latency and eliminates requests that return no new data; however, it requires a persistent connection, is less proxy-friendly than polling, and requires an additional client-side extension to integrate with HTMX.

=== HTMX

HTMX extends HTML with attributes that instruct the browser to fetch a URL and swap a portion of the page with the response.
No JavaScript is written; no build step is required.
The server returns an HTML fragment; the library inserts it into the designated element.
Forms continue to work without JavaScript — HTMX only enhances the default submit behaviour.

== Decision

The polling requirement (fetch a status fragment on a timer and replace the operation status element with the response) is expressed in a small number of HTML attributes with no JavaScript.
The server-rendered fragment model is consistent with Jinja2: the same template partials serve both full-page and fragment responses.

When the asyncio migration is complete, migrating from polling to SSE requires changing the server endpoint, loading the SSE client extension, and updating the attributes on the status element; the page structure does not change.

*Decision:* HTMX for the polling pattern; no other JavaScript framework.

== Consequences

* No JavaScript build toolchain is introduced.
* Forms degrade gracefully to standard HTML form submission when JavaScript is unavailable.
* Migrating to Server-Sent Events is a server-side change, an additional client-side extension, and attribute changes on the status element; the page structure is unaffected.
